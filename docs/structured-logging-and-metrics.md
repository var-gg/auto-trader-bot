# Structured Logging and Metrics

This document summarizes the minimum observability contract after the live/backtest split.

---

## Live structured log fields

Emit at least:
- `run_id`
- `slot`
- `command`
- `strategy_version`
- `decision_summary`
- `risk_reject_reason`
- `order_batch_id`
- `order_plan_id`
- `broker_request_id`
- `broker_response_id`

Optional but useful:
- latency ms
- market
- symbol count
- plan count
- skip count
- env / deploy version

Helper:
- `live_app.observability.structured_logging.build_live_run_log`

---

## Backtest structured log fields

Emit at least:
- `scenario_id`
- `data_range`
- `parameter_hash`
- `score_summary`
- `fill_summary`
- `strategy_version`
- `feature_version`
- `seed`

Optional but useful:
- trial number
- decision engine version
- universe size
- artifact path

Helper:
- `backtest_app.observability.structured_logging.build_backtest_run_log`

---

## Correlation guidance

To compare a live run with a backtest run, you need enough keys to answer:
- which run was this
- which strategy/version was active
- which plan/order batch was produced
- which broker exchange request/response corresponds to it
- which backtest scenario and parameter set is the closest counterpart

Correlation keys are more important than raw log volume.
Adding more logs without identifiers is not acceptable.

---

## Suggested metrics

### Live
- runs by slot
- command success/failure count
- risk reject count by reason
- plans generated by strategy version
- broker request latency / failure count

### Backtest
- scenarios executed
- trial count by study
- score distribution by strategy version
- fill ratio / unfilled ratio
- replay reproducibility failures

---

## Incident minimum

During any incident, logs should be enough to recover:
- affected `run_id`
- slot + command
- strategy version
- risk reject reason if any
- order_plan/order_batch id
- broker request/response id
- comparison backtest scenario id

If these cannot be recovered, observability is insufficient.
